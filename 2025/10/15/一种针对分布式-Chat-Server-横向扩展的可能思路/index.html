一种针对分布式 Chat Server 横向扩展的可能思路

水平扩展出发点

在可以横向简单扩展的架构中，最重要的是将 state 从要扩展的服务器中摘除，或者至少公开到可以快速查询的基建服务上，比如 Redis 节点上。
在此基础上，才可以允许 user 无论被负载均衡到哪一个实例上，都是等价的。

所以我的设想是，尽量将 state 转存到用户、Redis 外部节点上，让有需要的业务服务器自行查询指定 state。
这样，所有的服务器就可以是无状态的，只需要向注册中心注册，即可完成动态扩缩容和容灾调整。

状态分布分析

与 user session 相关的全局状态只有一个，就是 online_user 的信息，它会以 online:user_id -> gateway_addr 的方式存储于 Redis 当中。这是在

在以上流程中，除 Connector 外所有的业务服务器（即 Infrastructure 和 DataPersistence 以外的服务器），均不持有共享状态。

Connector 的状态

本质上 Connector 服务器的状态，是 Redis 中 online_users 状态的一部分，也即 Redis 是全局的 Connector 多实例状态的拷贝。

Connector 与 Redis 间状态的同步，可以通过带短过期时间的 key + 心跳机制实现。

  • key 过期删除

这是安全的，因为没有续约的 key 代表着用户已经超时未上线，删除其在线状态是合理的。

  • key 更新

在不限制用户单点登录的情况下，这是不安全的。但是考虑到用户的在线状态是记录在 Redis 中的，因此可以有两种选择：

① 禁止在线后新的异地上线。
② 新上线会顶替旧上线，但是这需要 Connector 也读取/监听 Redis 中的全局在线状态，实现稍麻烦。

只要实现了单点登录，更新就只会发生在一个用户重新连接的情况下，这种情景是安全的。不过，这需要 Connector 在一个用户断开连接时，立刻清理 local state 和 Redis state 中的 online_user，防止一个用户因为网络连接不稳，在 Redis state 还未过期时就重新连接，导致强制假掉线。（NOTICE：这似乎无法实现完美的一致性，即一断开就立刻完成 Redis 删除。）

Handler 的状态

Handler 是业务微服务器的统称，在这个最简化的 IM 系统中，它应该分为 User Service、Channel Service 以及最重要的 Message Service。
