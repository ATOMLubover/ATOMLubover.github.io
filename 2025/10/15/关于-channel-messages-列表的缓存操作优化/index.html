关于 channel:messages 列表的缓存操作优化

一、缓存失效的原因

  • 内存淘汰：由于内存占用过大，自动淘汰。
  • 超时淘汰：由于 key 达到了 TTL 时间点，自动淘汰。
  • DB 数据更新：cache 与 DB 值不一致。

二、缓存重建时可采用的 主动 操作

2.1 可能的方案

  • Cache Aside：业务层负责修改数据库，以及修改缓存。
  • Write Through：业务层均读写于缓存层，缓存层自动同步数据库。
  • Write-behind：业务层均读写于缓层存，随后由业务层/缓存层异步修改数据库。

2.2 业务层面临的 Cache Aside 的问题

  • 更新还是删除
  • 先操作数据库，还是先操作缓存
  • 原子性的事务性保证

三、需要处理的缓存失效问题

  • Cache Penetration：一个 key 不存在于 DB，导致无效请求反复抵达 DB。

  • Cache Breakdown：一个热点 key 突然过期，导致大量请求在缓存重建之前全部抵达 DB。

  • Cache Avalanche：多个 key 突然过期，导致导致大量请求在缓存重建之前全部抵达 DB。

3.1 Cache Penetration 解决方案

  • 设置对应 key 的空值（带短 TTL）

可以有效防止短时间内的大量请求进入数据库。

浪费 Redis 空间，但是仍然可控。

  • Bloom Filter

业务层直接拦截，更加高效，且适合分布式。

但是需要处理新 key 产生后的一致性。
且假如恰好一直访问一个 false positive 的值，仍然无法防护。

3.2 Cache Breakdown 解决方案

为了减少数据库压力，必然需要使用 singleflight，但是具体到阻塞时的操作，仍然有两种选择。

3.2.1 简单分布式锁

3.2.1-1 实现方法

当某个线程发现缓存失效时，使用 SET NX 尝试获取 Redis 中对应的 modify lock。

如果获取成功，则查询 DB，用 DB 数据重建缓存并返回，释放 modify lock。
如果获取失败，则指数退避尝试若干次 cache，在期限内获取到则返回 cache 数据，否则返回 503。当然，也可以多次失败强行读取 DB，但是如果这样，本质上并没有保护好 DB，如果 cache 长时间都未重建，可能本身就是 DB 超载。

3.2.1-1 优劣

优势：

  • 强一致性
  • value 内存消耗更小
  • 动态适应性更好

劣势：

  • 死锁、锁重入问题
  • 大量请求超时可能

3.2.2 分布式锁 + 逻辑过期

3.2.2-1 实现方法

修改前提：不设置 key 的 TTL，而是在 value 中存储 exp 字段。

当某个线程获取到 key，并且发现 exp 已经过期，则使用 SET NX 尝试获取 Redis 中对应的 modify lock。

如果获取成功，则单独开启一个协程进行 cache rebuild，随后立刻返回过期的 cahce 数据。
如果获取失败，立刻返回过期 cache 数据

3.2.2-2 优劣

优势：

  • 响应更快

劣势：

  • 低一致性
  • 整体内容消耗显著增多

Cache Avalanche 解决方案

其核心问题在于 多个 key 的过期时间集中在相近的时间内了，因此完全可以给这一系列的 key 的 TTL 设置为基准 TTL ± offset，其中 offset 是一个随机的振幅时间。

此外，也可以使用如下针对 单 key 有效的应对方法：

  • 使用 local cache 来防止服务全线崩溃
  • 利用 cache breakdown 中的 logical expiration，使用旧数据进行返回
